**Acceptance tests (for test-writer):**

Write a test file at `repo_man/test/repo_man/git_builder_test.exs`. Tag the module `@moduletag :integration` and `@moduletag :tmp_dir`. Each test calls `RepoMan.GitBuilder.build(tmp_dir, state)`, asserts the returned path exists as a git repo, then verifies the state using real `System.cmd` calls (not the application's `Git` module — raw git commands).

- `build/2` with `:clean` — returned path is a valid git repo (`File.dir?(Path.join(path, ".git"))`), `git status --porcelain` returns empty output, and the repo has at least one commit
- `build/2` with `:dirty` — returned path is a valid git repo, `git status --porcelain` returns at least one line
- `build/2` with `:behind` — returned path is a valid git repo, after `git fetch`, `git rev-list --count HEAD..origin/master` (or `origin/main`) returns a count > 0; ahead count is 0; working tree is clean
- `build/2` with `:diverged` — returned path is a valid git repo, after `git fetch`, both `git rev-list --count HEAD..origin/master` (behind) and `git rev-list --count origin/master..HEAD` (ahead) return counts > 0; working tree is clean
- `build/2` with `:topic_branch` — returned path is a valid git repo, `git branch --show-current` returns a branch name that is NOT `master` and NOT `main`; working tree is clean
- Each call returns a distinct path under `tmp_dir` (two sequential calls do not collide)
- No call touches any path outside the provided `tmp_dir`

**Edge cases:**

- `:behind` and `:diverged` states require a bare remote repo created under `tmp_dir`. The bare remote path must be a sibling of the clone, not nested inside it, to avoid `git init` confusion.
- `:behind` requires a `git fetch` to have run (so the clone's remote-tracking ref is updated) but the local branch must NOT have been merged — verify `{0, N}` not `{0, 0}`.
- `:diverged` requires local commits on the clone AND remote commits on the bare origin (via a separate pusher clone), plus a `git fetch` so the clone sees the remote divergence.
- Git identity (`user.email`, `user.name`) must be set via `git config` inside each repo before any `git commit` call — Docker has no global git config. This is the single most common failure mode in CI.
- The `@moduletag :tmp_dir` ExUnit tag creates the directory automatically and cleans it up after the test. GitBuilder receives the `tmp_dir` string from the test context and creates subdirectories within it.
- `:topic_branch` must use a branch name that is neither `master` nor `main` — use a fixed name like `"topic-branch"` to make assertions deterministic.

**Risks:**

- If `mix.exs` does not include `test/support` in `elixirc_paths` for the test env, the module won't compile in tests. Verify before dispatching: the scaffold (F02) standard Phoenix template includes this, but confirm.
- The `:behind` state builder must NOT call `git pull` or `git merge` after `git fetch` — that would change the state from `:behind` to `:clean`. Only `git fetch` is permitted.
- Naming collisions if multiple `build/2` calls use the same `tmp_dir` without unique subdirectory names. The implementation must use unique names per call (e.g., `"git-builder-#{state}-#{:erlang.unique_integer([:positive])}"`) to support multiple builds in a single test.

**Implementation details:**

- Single public function `build(tmp_dir, state)` pattern-matches on the five state atoms
- `:clean` -- `git init -b master`, configure identity, add README.md, commit. Returns clean repo path
- `:dirty` -- same as clean, then writes an untracked `dirty.txt` file
- `:behind` -- creates bare remote via temp init repo, clones it, pushes an extra commit to origin via a separate pusher clone, then `git fetch` on the clone (no merge/pull). Clone sees 1 behind, 0 ahead
- `:diverged` -- same remote/clone setup as behind, but also makes a local commit on the clone before fetching. Clone sees 1 behind, 1 ahead
- `:topic_branch` -- same as clean, then `git checkout -b topic-branch`
- All repos use `git init -b master` for consistent default branch naming (tests assert against `origin/master`)
- Git identity (`user.email`, `user.name`) configured in every repo that needs commits, including pusher clones
- Unique paths via `:erlang.unique_integer([:positive])` suffix per call, preventing collisions
- Private `git!/2` helper raises on non-zero exit codes with the command and output for debuggability
- No application modules used (no `RepoMan.Git`), only `System.cmd/3` and `File` operations
- Bare remote paths are siblings of clone paths (not nested), avoiding git init confusion
- Compiler warnings about `RepoMan.Git.Mock` are expected (Mox not yet defined; unrelated to this feature)
