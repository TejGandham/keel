**Acceptance tests (for test-writer):**

Group 1 — Source-level invariant (static): no `--force` in git.ex
- Verify that the source of `repo_man/lib/repo_man/git.ex` contains no occurrence of the string `--force`. Use `File.read!` on the known absolute path inside Docker (`/app/lib/repo_man/git.ex`) and assert the string is absent.

Group 2 — `--ff-only` always used on pull
- Verify that `Git.pull_ff_only/1` issues `--ff-only` by checking the source: `File.read!` the git.ex source and assert it contains `"--ff-only"`.
- Behavioral: build a `:behind` repo with `GitBuilder`, call `RepoMan.Git.pull_ff_only(path)`, assert result is `:ok`, assert the repo is now up-to-date (behind count is 0 after pull).
- Behavioral: build a `:diverged` repo, call `RepoMan.Git.pull_ff_only(path)`, assert `{:error, _}` is returned (fast-forward not possible), assert working tree is unchanged (ahead count still > 0).

Group 3 — Pull rejected when dirty
- Build a `:dirty` repo with `GitBuilder`.
- Manually construct a `%RepoStatus{}` matching the dirty state (dirty_count > 0, pull_eligible? false).
- Assert `RepoStatus.pull_eligible?` is false for a status with dirty_count > 0.
- Call `RepoMan.Git.pull_ff_only(path)` directly and verify the dirty files are still present after the call (working tree not modified).

Group 4 — Pull rejected when diverged
- Build a `:diverged` repo with `GitBuilder`.
- Assert `RepoStatus.pull_eligible?` is false for a status with ahead > 0 AND behind > 0.
- Call `RepoMan.Git.pull_ff_only(path)`, assert `{:error, _}` — ff-only fails as expected.
- Assert ahead count is still > 0 after the failed call.

Group 5 — Pull rejected when not on default branch
- Build a `:topic_branch` repo with `GitBuilder`.
- Assert `RepoStatus.pull_eligible?` is false for a status with `on_default?: false`.

Group 6 — Pull rejected when not behind (already up to date)
- Build a `:clean` repo with `GitBuilder` (clean = on default branch, 0 ahead, 0 behind).
- Assert `RepoStatus.pull_eligible?` is false for a status with behind == 0.

Group 7 — No working tree modification beyond ff-only pull
- Build a `:dirty` repo.
- Capture the dirty file list before any operation (`git status --porcelain`).
- Call `RepoMan.Git.fetch(path)`, assert `:ok`.
- Assert dirty file list is identical after fetch (fetch never touches working tree).

**Edge cases:**
- The source-level `--force` check must scan the full `git.ex` content, not just function bodies, to catch future additions anywhere in the file.
- `pull_ff_only/1` on a `:clean` repo that is already up-to-date returns `:ok` with "Already up to date." message from git — this is not an error. The source test covers correctness of the flag; no separate test needed here unless it aids clarity.
- The `RepoStatus.pull_eligible?` assertions work on struct values directly — no GenServer needed for Groups 3-6. Use `RepoStatus.new/1` (from `repo_man/lib/repo_man/repo_status.ex`) to build minimal structs for these assertions.
- For Groups 2-7, `GitBuilder` repos are local-only or bare-remote clones — all git operations run against `/tmp`, never touching `~/src/shred/`.

**Risks:**
- The source-level `--force` check uses a hardcoded path inside the Docker container (`/app/lib/repo_man/git.ex`). This is intentional per the project's Docker-first architecture but means the test is skipped or fails if run outside Docker. Tag it `@tag :integration` like all other real-git tests so it is excluded from the default `mix test` run and only runs under `mix test --include integration` (which is always inside Docker).
- `RepoServer` pull precondition enforcement is in `handle_cast(:pull, ...)` at `repo_man/lib/repo_man/repo_server.ex:156` — the guard checks `pull_eligible?: false`. The safety tests at Layer 1 test `RepoStatus.pull_eligible?` directly rather than round-tripping through the GenServer. This is sufficient per the spec: the invariant is that the derived field is correct. Layer 3 (F-series process behavior tests) already covers that RepoServer respects `pull_eligible?`.
- `async: true` is safe for all tests in this file because they use temp directories from `@moduletag :tmp_dir` and call `RepoMan.Git` directly (no shared process state, no Mox).
